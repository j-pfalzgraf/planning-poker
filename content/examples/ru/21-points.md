# 21 Story Points – Очень крупные изменения

> **Трудозатраты:** 1–2 недели
> **Риск:** Высокий
> **Тесты:** Комплексный набор тестов + QA-ревью

---

## Пример 1: Ролевые разрешения

**Название:** Реализация системы ролей и разрешений

**Описание:**
Введение ролей (Админ, Менеджер, Пользователь) с гранулярными разрешениями. UI-элементы и API-эндпоинты защищены соответственно.

**Критерии приёмки:**

- [ ] Роли: Админ, Менеджер, Пользователь, Гость
- [ ] Разрешения на функцию (CRUD)
- [ ] Админ-интерфейс для управления ролями
- [ ] Frontend: Условное отображение элементов
- [ ] Backend: Middleware для авторизации
- [ ] Аудит-лог изменений разрешений
- [ ] Миграция существующих пользователей

---

## Пример 2: Мультитенантность

**Название:** Введение поддержки мультитенантности

**Описание:**
Приложение должно поддерживать несколько независимых арендаторов (компаний). Строгое разделение данных.

**Критерии приёмки:**

- [ ] Tenant ID во всех релевантных таблицах
- [ ] Определение тенанта через субдомен или заголовок
- [ ] Переключатель тенантов для супер-админов
- [ ] Изолированные данные для каждого тенанта
- [ ] Конфигурация, специфичная для тенанта
- [ ] Миграция существующих данных

---

## Пример 3: Интернационализация (i18n)

**Название:** Полная мультиязычная поддержка (EN, DE, RU)

**Описание:**
Всё приложение должно быть доступно на трёх языках. Динамическое переключение языка без перезагрузки.

**Критерии приёмки:**

- [ ] Все UI-тексты экстернализированы
- [ ] Языковые файлы для EN, DE, RU
- [ ] Переключатель языка в хедере
- [ ] Язык сохраняется в настройках пользователя
- [ ] Форматы даты и чисел локализованы
- [ ] Подготовлена поддержка RTL

---

## Почему 21 балл?

- Широкие изменения по всей системе
- Влияние на архитектуру
- Высокий риск регрессий
- Сложная миграция
- Требуется интенсивное тестирование
